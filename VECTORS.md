# x402 v1.0 Canonical Test Vectors

Version: 1.0
Status: Frozen
Authority: Normative (defines interoperability)

---

## 1. What the vectors are

The file `testdata/x402-vectors-v1.json` contains the canonical test vectors for x402 v1.0. These vectors are normative. They define the exact byte-level behavior that any compliant implementation must reproduce.

The vectors are not examples. They are the interoperability contract between independent implementations. Two implementations that produce identical outputs for all vectors will interoperate correctly on the wire.

**If your implementation reproduces these vectors byte-for-byte, it is compliant with x402 v1.0.**

The vectors were generated by the Go reference implementation (`cmd/vecgen`) and independently verified by a Python implementation using only standard library functions. Both produce identical outputs for all 25 checks.

---

## 2. What is being verified

Each vector tests one or more of the following operations:

### Canonical JSON serialization

Challenge objects must be serialized deterministically before hashing. Serialization follows RFC 8785 (JSON Canonicalization Scheme):

- Object keys sorted lexicographically
- No whitespace outside strings
- Numbers in shortest decimal form (integers without decimal point)
- Strings in UTF-8 with standard JSON escaping

The `canonical_challenge_json` field in each vector contains the exact UTF-8 string that must be produced. The `canonical_challenge_hex` field contains the same bytes as lowercase hexadecimal.

### SHA-256 challenge hash

The challenge hash is computed as:

```
challenge_sha256 = lowercase_hex(SHA-256(utf8_bytes(canonical_challenge_json)))
```

This hash is carried in the proof as `challenge_sha256` and must match between client and server.

### base64url encoding

The `X402-Challenge` HTTP header value is:

```
base64url_no_padding(utf8_bytes(canonical_challenge_json))
```

base64url uses the URL-safe alphabet (`-` and `_` instead of `+` and `/`) with no `=` padding.

### Header-binding string

Request headers are canonicalized into a binding string per spec Section 4:

1. Select headers from the server's binding list
2. Lowercase each header name
3. Trim whitespace around each value
4. Sort by header name in ascending byte order
5. Concatenate as `name:value\n` (LF after each line, including the last)

Headers not present in the request are included with an empty value (e.g., `accept:\n`).

The `header_binding_string` field contains the exact UTF-8 string. The `headers_sha256` field contains `lowercase_hex(SHA-256(utf8_bytes(header_binding_string)))`.

### Body hashing

```
body_sha256 = lowercase_hex(SHA-256(raw_body_bytes))
```

For empty bodies: `SHA-256("")` = `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`.

The `body_bytes` field is hex-encoded raw bytes. The `body_sha256` field is the hash.

### Transaction identifier derivation

For Bitcoin-style transaction formats:

```
txid = lowercase_hex(byte_reverse(SHA-256(SHA-256(raw_tx_bytes))))
```

The double SHA-256 result (32 bytes) is byte-reversed before hex encoding. The `rawtx_hex` field contains the raw transaction as hex. The `txid` field contains the computed identifier.

### Reject conditions

Vectors with `"expected_result": "reject"` define cases where a compliant server must refuse the proof:

- **Expired challenge** -- `expires_at` is in the past
- **Invalid txid** -- `payment.txid` does not match the decoded transaction
- **Path mismatch** -- Proof bound to one path, request sent to another
- **Invalid version** -- `v` is 0 (unsupported)

## 3. How to use the vectors

These steps are language-agnostic. Implement them in your target language.

### Step 1: Load the vector file

Parse `testdata/x402-vectors-v1.json`. The top-level structure is:

```json
{
  "version": "1.0",
  "vectors": [ ... ]
}
```

### Step 2: For each vector, run applicable checks

**Challenge hash verification** (vectors with `canonical_challenge_json` and `challenge_sha256`):

1. Take the `canonical_challenge_json` string as UTF-8 bytes
2. Compute SHA-256
3. Encode as lowercase hexadecimal
4. Assert the result equals `challenge_sha256`

**Canonical JSON reproduction** (vectors with `canonical_challenge_json`):

1. Parse `canonical_challenge_json` as JSON into your language's native types
2. Re-serialize using your RFC 8785 implementation
3. Assert the output bytes are identical to the original `canonical_challenge_json`

**base64url verification** (vectors with `canonical_challenge_json` and `challenge_base64url`):

1. Take the `canonical_challenge_json` string as UTF-8 bytes
2. Encode with base64url (no padding)
3. Assert the result equals `challenge_base64url`

**Hex encoding verification** (vectors with `canonical_challenge_json` and `canonical_challenge_hex`):

1. Take the `canonical_challenge_json` string as UTF-8 bytes
2. Encode as lowercase hexadecimal
3. Assert the result equals `canonical_challenge_hex`

**Header hash verification** (vectors with `header_binding_string` and `headers_sha256`):

1. Take the `header_binding_string` as UTF-8 bytes
2. Compute SHA-256
3. Encode as lowercase hexadecimal
4. Assert the result equals `headers_sha256`

**Body hash verification** (vectors with `body_bytes` and `body_sha256`):

1. Decode `body_bytes` from hexadecimal to raw bytes
2. Compute SHA-256
3. Encode as lowercase hexadecimal
4. Assert the result equals `body_sha256`

**Transaction ID verification** (vectors with `rawtx_hex` and a 64-character `txid`):

1. Decode `rawtx_hex` from hexadecimal to raw bytes
2. Compute SHA-256(SHA-256(raw_bytes))
3. Byte-reverse the 32-byte result
4. Encode as lowercase hexadecimal
5. Assert the result equals `txid`

**Reject case verification** (vectors with `"expected_result": "reject"`):

1. Read the `notes` field to understand the rejection reason
2. Verify that your implementation would reject the described condition
3. For vectors with challenge data, verify the canonical hash is still reproducible (the challenge is valid; the rejection is based on context such as expiry or binding mismatch)

### Step 3: All checks must pass

A compliant implementation produces zero mismatches across all vectors. There is no partial compliance.

## 4. Reference verifier

The file `testdata/verify_vectors.py` is a minimal reference implementation of the verification procedure. It uses only Python standard library functions (no external dependencies).

```bash
python3 testdata/verify_vectors.py
```

Expected output:

```
VERDICT: PASS -- all 25 checks passed
The Python implementation reproduces all reference vectors exactly.
```

The Python verifier implements:

- RFC 8785 canonical JSON serialization (sorted keys, no whitespace, integer numbers)
- SHA-256 hashing
- base64url encoding (no padding)
- Double SHA-256 with byte reversal for txid
- All reject condition checks

Use it as a reference when implementing verification in another language.

The Go reference implementation consumes the same vectors in `internal/challenge/vectors_test.go`:

```bash
go test ./internal/challenge/ -run TestVectors -v
```

## 5. Compliance definition

An implementation of x402 v1.0 is compliant with the canonical test vectors if:

1. For every vector where `expected_result` is `"valid"` or `"reference"`, all computed values match the recorded values exactly. No deviations in encoding, hashing, or byte ordering.

2. For every vector where `expected_result` is `"reject"`, the implementation rejects the described condition. The specific error code or message is implementation-defined, but the proof must not be accepted.

3. The implementation's canonical JSON serialization, given the same input object, produces byte-identical output to the `canonical_challenge_json` field in each vector.

4. The implementation's SHA-256, base64url, and hex encoding functions produce identical outputs to the recorded values.

An implementation that satisfies all four conditions will interoperate with any other compliant implementation on the wire.

## 6. Versioning

These vectors are versioned as `x402-vectors-v1`. They correspond to x402 protocol specification version 1.0.

Rules:

- The vectors in `testdata/x402-vectors-v1.json` are frozen. The `generated_at` timestamp may change on regeneration, but all hashes, canonical bytes, and encoded values must remain identical.
- If a change to the implementation alters any vector output, the implementation has drifted from the specification. The implementation must be corrected, not the vectors.
- Future protocol versions will introduce new vector files (e.g., `x402-vectors-v2.json`). Existing vectors will not be modified retroactively.
- The vector generator (`cmd/vecgen`) can be re-run to verify that the implementation still produces the same outputs. If it produces different outputs, a regression has occurred.

To verify vector stability:

```bash
go run ./cmd/vecgen > /tmp/vectors-check.json
diff <(jq -S 'del(.generated_at)' testdata/x402-vectors-v1.json) \
     <(jq -S 'del(.generated_at)' /tmp/vectors-check.json)
```

No output means no drift.
